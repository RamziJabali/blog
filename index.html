    In this project my goal was to make API calls to a weather server, and populate the UI with several weather attributes.

    I started this program by first adding the dependencies required for Okhttp and Retrofit.

    Secondly I searched up a Youtube tutorial that would teach me the basics of Retrofit and Okhttp

    While I was learning the basics I wanted to be able to apply it to to the project I wanted to make, so I looked for a weather API to use in my program

    1. Learn Retrofit
    2. Learn Okhttp
    3. Get an API to apply what I learned

    Once I had written the API endpoint managers, I tried right away to make the API call. Once I ran the application, it crashed.

    After a couple of Google searches I realized that I was making the call on the main thread and that was the reason for the application crashing.

    I remember here is where I started to get into the discussion of thread management. I didn’t know what a thread in this context even really was, so I decided to Google it and I was bombarded with 100 different answers. Just for sanity sake I boiled it down to, ability to run multiple sequences of code at the same time using different threads(single sequential flow of control within a program).

    After that discovery I had to make an adjustment to the code to allow it to run on the background thread as opposed to the main thread. Running the API call in the main thread causes the application to crash because the UI also runs on it. The reason for the application crashing is because the API call might have to wait for a response, which causes the UI to to also wait which inturn causes it to freeze and crash.

    The solution to that was using enqueue, which asynchronously sends a request and gets a callback of the result. Which is executed on the background thread!

    Perfect, the application was up and running!

    After that I changed the architecture to MVVM, added test cases, and added GitHub actions, finally the application was a wrap!

    What did I learn

    1. Do NOT be intimidated by what you don’t know
    2. Do NOT worry about all the little details
    3. Get the application up and running, then worry about making it better

    This project is a Pokedex application that shows a set number of pokemon at first and allows you to click on a pokemon of your choosing to learn more about that character.

    Broken down into two pieces first Pokedex, a set of Pokemon, and Pokemon, a specified Pokemon from the Pokedex page.

    It will also allow you to scroll down till you reach the last Pokemon available in the poke.api

    The first step that I took was pickout an API that has adequate information about Pokemons

    Lucky for me there is a very well documented pokemon API Poke Api made by Paul Hallet that was perfect for this project.

    With that in place I first started by creating Koin dependency injections for Retrofit, OkHttp, and GsonConverterFactory so that I can run a quick test to see a sample of the pokedex api.

    After I got the network calls working I started to build the first UI component, a screen full of Pokemon characters with their names and pictures.

    Todo that I decided to go with the GridView which would allow me to show a good amount of Pokemon on one page.

    I was able to implement the GridView without much trouble, and was able to customize the GridView without much trouble either.

    Lucky for me I was able to find a different API that contained only images of the poekmons that corresponded with PokeAPI pokemon id numbers!

    I didn't know how I would be able to show these images in my application and that's when I discovered Picasso by Square which gave me a seamless experience incorporating the pokemon Images into the Pokedex UI!

    I first tried making the two UI elements into fragments as a step up in my coding game, but I had already made one UI element into an activity and the other into a fragment I was conflicted on whether or not to go the all fragment route or the all acitivity route.

    Since I had already made alot of progress using Activities as opposed to Fragments, I decided to recreate the fragment into a activity to keep the progress of the application steady.

    The next order of business was implementing the next UI element, a dedicated Pokemon page with multiple stats on the Pokemon.

    I needed one key piece of information to be passed to the Pokemon page, from the Pokedex page, which was the Pokemon ID.

    That gave me a bit of trouble, because I didn't know the proper etiquette of passing infromation between two activites.

    After researching and asking people that are knowledgable on the topic I ended up creating a newInstance static function that takes in Context and Pokemon Id integer.

    This newInstance function as the name implies returns a new instance of the Pokemon activity that is used to then launch it!

    After that was solved all that was left to do was to spit shine the code and make changes were they were needed and wanted.

    I ended up upgrading the Pokemon UI by adding horizontal progress bars in correlation to the pokemon stats and fixing text locations

    Another neat addition was adding animation to the launch and exit of my Pokemon Activity!

    A realized a number of problems while cleaning my code

    1. The pokemon image API was not working and had an incomplete list of the prokemons
    2. I was using the wrong endpoint for my Pokedex and had to make a huge number of calls to populate the Pokedex page
    3. No use cases were created for the Pokdex and Pokemon Page

    One point of conflict was realizing that I had been using the wrong endpoint and that I was making needless calls left and right inorder to create the Pokedex page, luckily it was a very simple change and didn't end up actually simplifying the application

    Replacing the pokemon image API was required a bit of finnese and turned out better than the previous API I was using.

    Creating usecases took a big burden of the viewmodel for the Pokedex and Pokemon pages and ended up reducing and making the code a lot prettier!

    Take away:

    1. Get your app up and running then start cleaning up and adding features to it!
    2. Don't be scared to delete your code in place for better code!
    3. Don't get cold feet, just because you're unsure of your capabilities!
